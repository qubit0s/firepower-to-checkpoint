# Firepower → Check Point migration toolkit

Convert a Cisco Firepower Threat Defense `show running-config` (Firepower 2110, FTD 7.2.x,
FXOS 2.12.x — ASA-style syntax) into Check Point objects, groups, services, access policy
and NAT on **Smart-1 Cloud**, using the `check_point.mgmt` Ansible collection.

The parser never touches Check Point. All changes happen through the playbooks.

## Quickstart (5 steps)

```bash
# 1. Clone
git clone <your-repo-url> firepower-to-checkpoint
cd firepower-to-checkpoint

# 2. Setup  (creates .venv, installs ciscoconfparse2 + Ansible + the
#            check_point.mgmt collection)
./1_setup.sh

# 3. Edit inventory/hosts.ini  -> mgmt host + ONE auth block
#    Smart-1 Cloud (API key + cloud id) OR on-prem (API key, or user/password)
#    (see "Connecting to Check Point" below)

# 4. Parse your Firepower 'show running-config'
./2_parse.sh /path/to/show-running-config.txt
#    then REVIEW vars/*.yml  and  vars/_review_unsupported.yml

# 5. Apply  (does NOT publish by default -- review & publish in SmartConsole)
./3_apply.sh              # full migration in ONE session (objects->...->nat)
./3_apply.sh objects -C   # dry-run a single stage first (check mode)
```

That's it. `2_parse.sh` and `3_apply.sh` use the `.venv` automatically — no manual activation.
By default nothing is published: the run leaves one Check Point session for you to review and
publish manually (see "Publishing & review").

## Connecting to Check Point (step 3)

`inventory/hosts.ini` ships as a secret-free template that works for **Smart-1 Cloud** and
for **on-premise** Security Management (open server or appliance). Set the management host,
then uncomment exactly one auth block:

- **(A) Smart-1 Cloud** — `ansible_api_key` + `ansible_cloud_mgmt_id`. Host is
  `<tenant>.maas.checkpoint.com`. Get the API key from *Infinity Portal → Global Settings →
  API Keys → service "Smart-1 Cloud"*, and the cloud id from *Smart-1 Cloud → Settings → General*.
- **(B) On-premise, API key** — `ansible_api_key` only. Host is the management IP/hostname.
  Generate the key in *SmartConsole → Manage & Settings → Permissions & Administrators →
  Administrators → New → Authentication Method: API Key*.
- **(C) On-premise, username/password** — `ansible_user` + `ansible_password`.

`ansible_cloud_mgmt_id` is **only** for Smart-1 Cloud; omit it on-prem. The template has
placeholders only — keep your filled-in clone temporary and rotate the key afterward so
secrets are never committed. (Optional hardening: put secrets in an encrypted
`group_vars/all.vault.yml` via `ansible-vault`.)

## Publishing & review

By default the playbooks **do not publish** (`cp_publish: false` in
`inventory/group_vars/all.yml`). They create everything, then leave it for you to review in
SmartConsole under **Manage & Settings → Sessions** and publish manually.

Because Check Point sessions are private until published, unpublished changes are visible
**only within the session that made them**. So for an unpublished run, use the single-session
orchestrator:

```bash
./3_apply.sh            # site.yml: all stages in ONE session, nothing published
```

Running the per-stage playbooks separately opens a **new session each time**, so a later
stage can't see an earlier stage's *unpublished* objects. That's fine if you publish each
stage — set `cp_publish: true` (each stage publishes at its end), or publish a stage manually
before running the next. To auto-publish the whole single-session run, set `cp_publish: true`
and run `./3_apply.sh`.

## Manual usage (if you prefer not to use the wrappers)

```bash
source .venv/bin/activate
python parser/ftd_to_cp.py --config /path/to/show-running-config.txt --out vars
ansible-playbook playbooks/1_objects.yml    # or site.yml; add -C for dry-run
```

**Order matters:** groups, policy and NAT reference objects, so `objects` must run before
`object-groups`/`services`, and all of those before `policy` and `nat`. `site.yml` runs all
five stages in that order within a single session (`./3_apply.sh` with no stage). The per-stage
playbooks (`1_objects.yml` … `5_nat.yml`) each open their own session — see "Publishing & review"
for how that interacts with unpublished changes.

## What didn't make it in — two reports

Nothing is dropped silently. Two report files tell you what to look at:

- **`vars/_review_unsupported.yml`** (parse time) — Cisco constructs the parser could not
  translate to a Check Point object (e.g. an unknown protocol keyword). Only written when
  there's something to review; absent means the parse was clean.
- **`reports/<stage>_not_imported.yml`** (apply time) — items that reached Check Point but
  were **rejected** by the API, each with the error. The create tasks use `ignore_errors`, so
  one bad item never aborts the run; the rest still import and the failures are recorded here.
  Written every run (per stage), with `failed_count: 0` and an empty list when all imported.

So the review flow is: after `2_parse.sh`, check `_review_unsupported.yml`; after `3_apply.sh`,
check `reports/`.

## Cisco → Check Point mapping

| Cisco FTD (ASA syntax) | Check Point module | Notes |
|---|---|---|
| `object network … host` | `cp_mgmt_host` | |
| `object network … subnet` | `cp_mgmt_network` | dotted mask → `mask_length` |
| `object network … range` | `cp_mgmt_address_range` | |
| `object network … fqdn` | `cp_mgmt_dns_domain` | name prefixed with `.`, `is_sub_domain: false` |
| `object-group network` | `cp_mgmt_group` | created empty, then populated (handles nesting) |
| `object service tcp/udp` | `cp_mgmt_service_tcp` / `_udp` | `destination eq/range` |
| `object-group service` | `cp_mgmt_service_group` | `tcp-udp` groups expand to both protocols |
| `object-group protocol` | `cp_mgmt_service_group` of `cp_mgmt_service_other` | each protocol → "Other" service by IP-protocol number (GRE=47, ESP=50, OSPF=89…); `ip` → `Any` |
| `access-list … extended` | `cp_mgmt_access_rule` | `permit`→Accept, `deny`→Drop; written to `cp_access_layer` in ACL order |
| object NAT `static` | `cp_mgmt_nat_rule` method `static` | |
| object NAT `dynamic` | `cp_mgmt_nat_rule` method `hide` | `dynamic interface` → hide behind `cp_hide_behind_object` |
| manual/twice NAT | `cp_mgmt_nat_rule` | source/destination/service mapped; `interface` → gateway |

**Inline literals.** Where Cisco embeds a bare address or port (e.g. `network-object host
10.0.0.1`, `eq https`), Check Point has no name to reference, so the parser synthesises a
deterministic named object (`h_10.0.0.1`, `n_…`, `svc_tcp_443`) and adds it to the right
vars file so the dependency exists before the group/rule that uses it.

## Known limitations (review these)

- **NAT rules are keyed by name (R81+).** `cp_mgmt_nat_rule` requires a unique `name` on
  R81+ management, so the parser assigns each rule one (`objnat-…`, `manualnat-…`) and
  re-running updates rather than appends; `position` controls ordering only. Still review
  the NAT rulebase after the first apply.
- **`object-group protocol`** becomes a `cp_mgmt_service_group` whose members are
  `cp_mgmt_service_other` objects (one per IP protocol, by number). Unknown protocol keywords
  not in the parser's protocol map are flagged in `_review_unsupported.yml`.
- **Predefined service names** (`Any`, and the icmp service in `cp_icmp_service`, default
  `icmp-proto`) must exist on your management. Adjust `cp_icmp_service` in `inventory/group_vars/all.yml`
  if your version names it differently (e.g. `echo-request`).
- **Hide-NAT target.** `cp_hide_behind_object` (default `Internet-GW`) must be the real
  Check Point gateway object that replaces the Firepower.
- **Unknown port names** not in the parser's port map are passed through verbatim and flagged
  in `_review_unsupported.yml`.

## Validation performed

- Parser runs on `samples/ftd_running-config.txt`; output is deterministic across re-runs.
- All generated `vars/*.yml` and `inventory/group_vars/all.yml` are well-formed YAML.
- All six playbooks pass `ansible-playbook --syntax-check`.
- The two non-trivial Jinja transforms (icmp service remap; `interface`→gateway for NAT)
  were rendered against the generated vars and produce the expected values.
- Every parameter used in the playbooks was verified against the real `check_point.mgmt`
  module argument specs (collection source), including `cp_mgmt_service_other` used for
  protocol groups. This caught one required field — NAT rules need a unique `name` on R81+ —
  which is now passed from the parser-generated names.
- The apply-time failure recorder (`tasks/record_not_imported.yml`) was tested with a seeded
  failure and correctly writes `reports/<stage>_not_imported.yml` with the object and error.

## Linting

The repo is kept lint-clean. To check:

After `1_setup.sh`, the venv already has `yamllint` and `ansible-lint` (bundled in
`ansible-dev-tools`); only `ruff` and `shellcheck` are extra.

```bash
source .venv/bin/activate
pip install ruff                                # only extra Python linter needed
ruff check parser/                              # Python (or: pyflakes + pycodestyle)
yamllint .                                      # YAML (uses .yamllint in repo)
ansible-lint playbooks/                         # Ansible (production profile)
shellcheck *.sh                                 # shell wrappers (install separately)
```

Current status: `ruff`/`pyflakes`/`pycodestyle`, `yamllint`, and `ansible-lint` (production
profile) all pass. The parser emits yamllint-clean vars (document start + indented sequences).
